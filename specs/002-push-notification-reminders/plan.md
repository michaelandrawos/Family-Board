# Implementation Plan: Push Notification Reminders

**Branch**: `002-push-notification-reminders` | **Date**: 2026-04-03 | **Spec**: [spec.md](./spec.md)
**Input**: Feature specification from `/specs/002-push-notification-reminders/spec.md`

## Summary

Add push notification reminders for calendar tasks to Family Board. When a task has a time
set, the user can choose a reminder offset (at time / 15 / 30 / 60 min before). The app
registers an inline Service Worker (code embedded in `index.html`, registered via Blob URL)
that fires a `showNotification()` call at the scheduled time. Notification permission is
requested on first Calendar screen open. Clicking a notification opens the app to the
Calendar screen. All changes remain within `index.html`.

## Technical Context

**Language/Version**: HTML5 / CSS3 / ES6+ — no transpilation, no build step
**Primary Dependencies**: Web Notifications API, Service Worker API (both browser-native)
**Storage**: `data.tasks[i].reminder` (number | null) synced via JSONBin; SW registration is device-local
**Testing**: Manual testing in Chrome Android / iOS Safari PWA
**Target Platform**: iOS Safari PWA (primary); Chrome Android (secondary)
**Project Type**: Single-file web app — all code including Service Worker in `index.html`
**Performance Goals**: Notification arrives ≤ 60 seconds of scheduled time
**Constraints**: All code in `index.html`; SW registered via Blob URL; no server-side push
**Scale/Scope**: ~30 tasks max; 2 devices; independent per-device scheduling

## Phase 0: Research

*See [research.md](./research.md) for full findings.*

Key decisions:
- **Blob URL SW registration**: `navigator.serviceWorker.register(URL.createObjectURL(new Blob([swCode], {type:'application/javascript'})))` — supported in all modern browsers; keeps single-file
- **SW notification scheduling**: Use `setTimeout` inside SW via `postMessage` for precise firing; SW `onnotificationclick` opens the app to the correct date
- **Notification ID**: Use task ID as the notification tag to allow cancellation via `reg.getNotifications({tag})` then `n.close()`
- **iOS caveat**: iOS 16.4+ supports SW notifications in PWA mode only; graceful degradation for non-PWA
- **Permission timing**: Request on first `renderCal()` call if not yet asked; use `Notification.permission` guard

### New / modified functions in `<script>`

| Function | Purpose |
|---|---|
| `const SW_CODE` | String constant containing Service Worker source code |
| `registerSW()` | Creates Blob URL, registers SW once, stores ref in `swReg` |
| `scheduleNotification(task)` | Posts message to SW with task name, fire time, tag |
| `cancelNotification(taskId)` | Gets notification by tag and closes it; cancels pending |
| `requestNotifPermission()` | Calls `Notification.requestPermission()` if not yet asked; shows denied banner if denied |
| `renderPermBanner()` | Shows/hides the permission-denied banner in Calendar screen |
| (extended) `addDayTask()` | Reads reminder offset from modal; stores in task; calls `scheduleNotification` |
| (extended) `saveEdit()` | Reads reminder offset from edit panel; cancels old, schedules new notification |
| (extended) `delTask()` | Calls `cancelNotification(task.id)` before splicing |
| (extended) `renderCal()` | Calls `requestNotifPermission()` on first render |

### Service Worker responsibilities

The inlined SW handles:
- `message` event: receives `{type:'SCHEDULE', tag, title, body, fireTime}`, uses `setTimeout` to call `self.registration.showNotification()`
- `message` event: receives `{type:'CANCEL', tag}`, calls `self.registration.getNotifications({tag})` then closes
- `notificationclick` event: calls `clients.openWindow(url + '?date=YYYY-MM-DD')` to open app to relevant date
